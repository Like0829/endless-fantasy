## 搭建前准备：创建会话变量

### 操作路径

在 Dify Chatflow 编辑器中：

```
右上角 → 点"变量"按钮 → 弹出面板
     → 选"会话变量"选项卡
     → 点"添加变量"按钮
     → 逐个创建以下变量
```

### 变量列表

每个变量创建3个字段：**变量名**、**类型**、**默认值**。按以下表格逐个添加：

| # | 变量名 | 类型 | 默认值 | 说明 |
|---|--------|------|--------|------|
| 1 | `player_name` | String | `调查员` | 玩家名称 |
| 2 | `hp` | Number | `100` | 当前生命值 |
| 3 | `max_hp` | Number | `100` | 最大生命值 |
| 4 | `san` | Number | `100` | 当前理智值 |
| 5 | `max_san` | Number | `100` | 最大理智值 |
| 6 | `strength` | Number | `10` | 力量 3-18 |
| 7 | `dex` | Number | `10` | 敏捷 3-18 |
| 8 | `pow` | Number | `10` | 意志 3-18 |
| 9 | `con` | Number | `10` | 体质 3-18 |
| 10 | `intelligence` | Number | `10` | 智力 3-18 |
| 11 | `current_location` | String | `小镇入口` | 当前主场景 |
| 12 | `current_sub_location` | String | `` | 当前子场景（留空） |
| 13 | `inventory` | String | `手电筒,笔记本,急救包` | 携带物品，逗号分隔 |
| 14 | `clues` | String | `` | 已发现线索（留空） |
| 15 | `is_dead` | Boolean | `false` | 是否死亡 |
| 16 | `is_crazy` | Boolean | `false` | 是否疯狂 |
| 17 | `llm_raw_output` | String | `` | LLM原始输出（桥接变量，4个LLM共用） |

### 关键要点

- **变量名必须完全一致**（包括大小写、下划线），否则后续代码节点会找不到变量
- 所有变量必须设置为 **"可读写"**（默认就是可读写）
- 创建顺序无所谓，但建议按上面顺序创建方便核对
- 创建完检查列表是否有17个变量

## 节点流程图总览

以下是用文本模拟的完整节点连接图，搭建时对照此图连接：

```
[开始] ──→ [代码节点①: 读取会话变量] ──→ [HTTP: 获取世界状态] ──→ [代码节点②: 路由器]
                                                                              │
                                                                              ▼
                                                              ┌── IF/ELSE: region=="town"? ── True ──→ [LLM-城镇区] ──→ [变量赋值: 存LLM输出] ──┐
                                                              │                                                                                  │
                                                              └── False ──→ ┌── IF/ELSE: region=="coastal"? ── True ──→ [LLM-海滨区] ──→ [变量赋值: 存LLM输出] ──┐
                                                                           │                                                                                        │
                                                                           └── False ──→ ┌── IF/ELSE: region=="dark"? ── True ──→ [LLM-禁区] ──→ [变量赋值: 存LLM输出] ──┐
                                                                                        │                                                                               │
                                                                                        └── False ──→ [LLM-通用] ──→ [变量赋值: 存LLM输出] ──────────────────────────────┘
                                                                                                                                                                        │
                                                                                                                                                                        ▼
                                                                                                                                 [代码节点③: 解析输出] ← 读取 llm_raw_output
                                                                                                                                        │
                                                                                                                                        ▼
                                                                                         ┌── IF/ELSE: need_update? ── True ──→ [HTTP: 更新世界状态] ──┐
                                                                                         │                                                              │
                                                                                         └── False ──────────────────────────────────────────────────────┤
                                                                                                                                                        ▼
                                                                                                                                          [代码节点④: 计算新HP/SAN]
                                                                                                                                                  │
                                                                                                                                                  ▼
                                                                                                                                          [变量赋值: 更新状态]
                                                                                                                                                  │
                                                                                                                                                  ▼
                                                                                                                                          [直接回复: Answer]
```

> **关键连接规则**：
> - 4个LLM节点的输出先连到各自的 **"变量赋值"节点**（将LLM输出写入 llm_raw_output）
> - 4个"变量赋值"节点**都连接到**节点L（代码节点③ — 解析输出）
> - 节点L从**会话变量 llm_raw_output** 读取文本，不再直接引用LLM节点
> - 节点N（更新世界）和节点M的False分支**都连接到**节点O1（代码节点④）
> - 节点O1连接到节点O2（变量赋值），节点O2连接到节点P（Answer）

## 节点A：开始节点

### 位置

在 Dify Chatflow 编辑器中，**开始节点**是画布上的第一个节点，默认已存在。

### 配置内容

点击开始节点，在右侧面板中：

**1. 对话开场白**（可选，建议设置）：

在"对话开场白"输入框中写入：

```
迷雾笼罩着阿卡姆小镇，你站在小镇入口的石板路上。空气中的咸腥味告诉你——海就在不远处。你能看到远处的建筑轮廓在雾中若隐若现：一座哥特式教堂的尖顶、一座破败的灯塔、还有一栋两层木屋。脚下的石板路布满青苔，踩上去有些湿滑。

这座小镇藏着什么秘密？你握紧手中的手电筒，深吸一口气，踏入了迷雾之中。
```

**2. 变量设置**：开始节点默认没有需要配置的变量，直接下一步。

**3. 输出连接**：将开始节点的输出点连接到 **节点B（代码节点①）**。

### 节点连线操作

在画布上：
1. 鼠标悬停在开始节点的右侧圆点上
2. 按住鼠标左键
3. 拖出一条线到目标节点的左侧圆点
4. 松开鼠标完成连接

#### 第4步：复制代码

将以下代码完整复制到代码编辑区：

```python
import random


def main(player_name: str, hp: int, max_hp: int, san: int, max_san: int,
         strength: int, dex: int, pow: int, con: int, intelligence: int,
         current_location: str, current_sub_location: str,
         inventory: str, clues: str,
         is_dead: bool, is_crazy: bool) -> dict:
    """
    将所有会话变量组合，生成角色状态文本
    用于注入到LLM的系统提示词中
    """
    # ===== 生成真实随机骰值（解决LLM骰值重复问题） =====
    # 每次聊天产生一组新随机数，LLM必须按顺序使用
    d100_pool = [random.randint(1, 100) for _ in range(20)]
    d20_pool = [random.randint(1, 20) for _ in range(10)]
    dice_seed = random.randint(10000, 99999)
    dice_pool_text = (
        f"【本次骰值池】（种子:{dice_seed}，每次对话不同，请按顺序从左到右使用，用完循环）\n"
        f"  d100骰值: {', '.join(str(v) for v in d100_pool)}\n"
        f"  d20骰值: {', '.join(str(v) for v in d20_pool)}"
    )

    # 构建技能值（基于属性计算）
    # 每个技能 = 基础值 + 属性加成，上限99%
    skills = {
        "侦察": min(99, 25 + intelligence * 5),
        "聆听": min(99, 20 + pow * 5),
        "图书馆使用": min(99, 20 + intelligence * 5),
        "神秘学": min(99, 5 + pow * 5),
        "潜行": min(99, 20 + dex * 5),
        "说服": min(99, 15 + pow * 5),
        "格斗": min(99, 20 + (strength + dex) * 2),
        "闪避": min(99, dex * 4),
        "急救": min(99, 10 + intelligence * 4 + dex * 2),
        "追踪": min(99, 10 + intelligence * 4 + pow * 2),
    }

    # 构建完整角色数据对象
    role_state = {
        "player_name": player_name,
        "hp": hp,
        "max_hp": max_hp,
        "san": san,
        "max_san": max_san,
        "strength": strength,
        "dex": dex,
        "pow": pow,
        "con": con,
        "intelligence": intelligence,
        "current_location": current_location,
        "current_sub_location": current_sub_location,
        "inventory": inventory,
        "clues": clues,
        "is_dead": is_dead,
        "is_crazy": is_crazy,
        "skills": skills
    }

    # 计算HP/SAN状态描述（用//代替int()避免参数名int冲突）
    hp_pct = hp * 100 // max_hp if max_hp > 0 else 0
    san_pct = san * 100 // max_san if max_san > 0 else 0

    if hp_pct > 70:
        hp_status = "健康"
    elif hp_pct > 40:
        hp_status = "轻伤"
    elif hp_pct > 20:
        hp_status = "重伤"
    else:
        hp_status = "濒死"

    if san_pct > 70:
        san_status = "稳定"
    elif san_pct > 40:
        san_status = "不安"
    elif san_pct > 20:
        san_status = "恍惚"
    else:
        san_status = "濒临崩溃"

    # 格式化技能列表文本
    skill_lines = "\n".join([f"  - {k}: {v}%" for k, v in skills.items()])

    # 子场景（如果非空）
    sub_loc_line = f"子场景：{current_sub_location}" if current_sub_location else ""

    # 构建最终的可读文本
    role_state_text = f"""===== 角色状态 =====
玩家名称：{player_name}
━━━━━━━━━━━━━━━━
【生命值】HP: {hp}/{max_hp}（{hp_status}）
【理智值】SAN: {san}/{max_san}（{san_status}）
━━━━━━━━━━━━━━━━
【属性】
  力量(STR): {strength}  敏捷(DEX): {dex}
  意志(POW): {pow}  体质(CON): {con}
  智力(INT): {intelligence}
━━━━━━━━━━━━━━━━
【技能】
{skill_lines}
━━━━━━━━━━━━━━━━
当前位置：{current_location}
{sub_loc_line}
携带物品：{inventory}
已发现线索：{clues if clues else "无"}
死亡状态：{"是" if is_dead else "否"}
疯狂状态：{"是" if is_crazy else "否"}
=====================

{dice_pool_text}"""

    return {
        "role_state": role_state,
        "role_state_text": role_state_text,
        "current_location": current_location,
        "dice_pool_text": dice_pool_text,
        "d100_1": d100_pool[0],
        "d100_2": d100_pool[1],
        "d100_3": d100_pool[2],
    }
```

#### 第5步：确认输出变量

代码中的 `return` 语句会定义输出变量。Dify会自动识别：

| 输出变量 | 类型 | 说明 |
|---------|------|------|
| `role_state` | Object | 完整角色数据（给其他代码节点用） |
| `role_state_text` | String | 格式化文本（含骰值池，给LLM节点用） |
| `current_location` | String | 当前地点（给路由器用） |
| `dice_pool_text` | String | 预生成骰值池文本 |
| `d100_1` | Number | 第一个预生成d100值 |
| `d100_2` | Number | 第二个预生成d100值 |
| `d100_3` | Number | 第三个预生成d100值 |

> 新增的 `dice_pool_text` 已自动拼接到 `role_state_text` 末尾，LLM节点只需引用 `{{body.role_state_text}}` 即可同时获取角色状态和骰值池。

在右侧面板的"输出变量"区域，Dify会自动显示这三个变量，不需要手动填写。

#### 第6步：测试节点（可选）

点击右下角的"运行"按钮（▶），可以测试这个节点。在弹出的测试窗口中：

1. 系统会自动用默认值填充所有输入变量
2. 点击"运行"查看输出
3. 检查输出的 `role_state_text` 是否格式正确

#### 第7步：连接下一个节点

从节点B右侧圆点拖线连接到 **节点C（HTTP请求）**。

## 节点C：HTTP请求 — 获取世界状态

### 在Dify中的操作步骤

#### 第1步：创建节点

```
从节点B右侧圆点拖出线 → 松开 → 选择"HTTP请求"节点
```

#### 第2步：配置基本信息

| 字段 | 填什么 |
|------|--------|
| 节点名称 | `获取世界状态` |
| 方法 | 选 `GET` |
| URL | `http://192.168.88.100:8000/world-state` |

#### 第3步：请求体类型（Body）

在 Dify 的 HTTP 节点中，必须先指定 Body 类型，**GET 请求选 `None`**：

```
在右侧面板找到 Body 选项卡
→ 点开下拉菜单
→ 选择 "None"
```

> **注意**：Dify 的 HTTP 节点默认 Body 类型不是 None，如果这里不选，GET 请求可能会带空 Body 导致请求异常。

#### 第4步：请求头设置

默认情况下，GET请求不需要特殊请求头。但如果你需要，可以添加：

```
点击"添加请求头" → 
  键: Content-Type
  值: application/json
```

（对于GET请求，这一步可省略）

#### 第5步：超时设置

| 字段 | 值 |
|------|-----|
| 超时（秒） | `15` |

#### 第6步：响应处理

Dify会自动解析HTTP响应的JSON为 `body` 变量。

**响应格式示例**（你的后端返回的数据）：

```json
{
  "atmosphere": 50,
  "atmosphere_description": "迷雾笼罩，空气中弥漫着不安的气息"
}
```

在后续LLM节点中通过 `{{body}}` 引用此数据。

#### 第7步：测试节点

点击"运行"测试，确认能成功返回数据。如果测试失败：

- 检查后端服务是否运行（`http://192.168.88.100:8000` 是否能访问）
- 检查URL路径是否正确（`/world-state`）
- 检查后端日志是否有错误

#### 第7步：连接下一个节点

从节点C右侧圆点拖线连接到 **节点D（代码节点②）**。

## 节点D：代码节点② — 路由器

### 功能说明

这个节点根据玩家当前所在位置（current_location），判断玩家在哪个区域，返回一个region值（town/coastal/dark/general），后面的IF/ELSE节点根据region值路由到对应的LLM。

### 在Dify中的操作步骤

#### 第1步：创建节点

```
从节点C右侧圆点拖出线 → 松开 → 选择"代码节点"
```

```text
## 一、检定系统（核心机制）

当玩家执行有风险的动作时，你需要进行"检定"来决定结果。

### 1.1 检定规则

检定流程：
1. 确定使用的技能或属性
2. 计算成功率（基于技能值或属性）
3. 从角色状态文本末尾的`骰值池`中取下一个d100值作为本次骰值（从左到右按顺序使用）
4. 骰值 ≤ 成功率 → 成功
5. 骰值 > 成功率 → 失败
6. 骰值 ≤ 成功率的1/5 → 大成功（极好效果）
7. 骰值 = 100 或 > 成功率的95% → 大失败（灾难性后果）

> ⚠️ 必须使用角色状态文本末尾`【本次骰值池】`中的d100值！每个检定消耗一个，用完回到开头循环。不允许自己编造骰值。

### 1.2 属性检定公式

属性值 × 5 = 基础成功率（例如STR 14 → 70%）

- STR 力量：推开重物、攀爬、近战伤害加成
- DEX 敏捷：闪避、潜行、平衡、先手顺序
- CON 体质：抵抗毒素/疾病、持续奔跑
- INT 智力：推理、解谜、记忆、理解复杂信息
- POW 意志：抵抗恐惧、说服他人、施展/抵抗法术

### 1.3 难度修正

- 有利条件（工具、环境优势）：+10% 到 +30%
- 不利条件（黑暗、紧张、受伤）：-10% 到 -30%
- 极度不利（狂风暴雨、重伤）：-40% 到 -60%

### 1.4 对抗检定

当玩家与NPC/怪物对抗时：双方各自投骰，都成功则成功率更高者胜，一方成功则成功者胜，都失败则维持现状。

## 二、战斗系统

战斗流程：先手判定(DEX对抗) → 攻击方投格斗 → 防守方可选闪避 → 命中则投伤害 → 描述结果

武器伤害：
- 徒手1d3、匕首1d4+1(可投掷)、撬棍1d6(可撬门)、棍棒1d6
- 猎刀1d6+1、手枪1d8(需火器技能)、猎枪1d10(装填慢)、临时武器1d4

伤害规则：
- HP ≤ 0：濒死，每回合继续损失1HP
- HP ≤ -max_hp：死亡
- 同伴可在3回合内急救稳定伤势

大成功（骰值≤成功率/5）：战斗双倍伤害/缴械，侦察发现额外信息，说服态度大幅提升
大失败（骰值=100或>成功率95%）：战斗武器脱手/自伤，潜行制造声响，交涉激怒对方
```

```text
# 完整游戏规则

## 一、检定系统（核心机制）

当玩家执行有风险的动作时，你需要进行"检定"来决定结果。

### 1.1 检定规则

检定流程：
1. 确定使用的技能或属性
2. 计算成功率（基于技能值或属性）
3. 从角色状态文本末尾的`骰值池`中取下一个d100值作为本次骰值（从左到右按顺序使用）
4. 骰值 ≤ 成功率 → 成功
5. 骰值 > 成功率 → 失败
6. 骰值 ≤ 成功率的1/5 → 大成功（极好效果）
7. 骰值 = 100 或 > 成功率的95% → 大失败（灾难性后果）

> ⚠️ 必须使用角色状态文本末尾`【本次骰值池】`中的d100值！每个检定消耗一个，用完回到开头循环。不允许自己编造骰值。

### 1.2 属性检定公式

属性值 × 5 = 基础成功率（例如STR 14 → 70%）

- STR 力量：推开重物、攀爬、近战伤害加成
- DEX 敏捷：闪避、潜行、平衡、先手顺序
- CON 体质：抵抗毒素/疾病、持续奔跑
- INT 智力：推理、解谜、记忆、理解复杂信息
- POW 意志：抵抗恐惧、说服他人、施展/抵抗法术

### 1.3 难度修正

- 有利条件（工具、环境优势）：+10% 到 +30%
- 不利条件（黑暗、紧张、受伤）：-10% 到 -30%
- 极度不利（狂风暴雨、重伤）：-40% 到 -60%

### 1.4 对抗检定

当玩家与NPC/怪物对抗时：双方各自投骰，都成功则成功率更高者胜，一方成功则成功者胜，都失败则维持现状。

## 二、战斗系统

战斗流程：先手判定(DEX对抗) → 攻击方投格斗 → 防守方可选闪避 → 命中则投伤害 → 描述结果

武器伤害：
- 徒手1d3、匕首1d4+1(可投掷)、撬棍1d6(可撬门)、棍棒1d6
- 猎刀1d6+1、手枪1d8(需火器技能)、猎枪1d10(装填慢)、临时武器1d4

伤害规则：
- HP ≤ 0：濒死，每回合继续损失1HP
- HP ≤ -max_hp：死亡
- 同伴可在3回合内急救稳定伤势

大成功（骰值≤成功率/5）：战斗双倍伤害/缴械，侦察发现额外信息，说服态度大幅提升
大失败（骰值=100或>成功率95%）：战斗武器脱手/自伤，潜行制造声响，交涉激怒对方

## 三、理智系统（SAN）

恐怖等级：
- 轻微恐怖：0/1d3（看到尸体、诡异声音）
- 中等恐怖：1/1d6（超自然现象、遭遇怪物）
- 强烈恐怖：1d3/1d10（直面神话生物、同伴惨死）
- 极致恐怖：1d6/2d10（目击不可名状之神、禁忌古籍）

临时疯狂（单次SAN损失≥5，投1d10）：
1-2:失忆 3-4:惊恐逃跑 5-6:暴力狂乱 7-8:幻觉 9-10:昏厥

SAN恢复：安全休息1d6、解谜推进1d3、镇静剂1d6、同伴安慰1d3、逃离危险1d3

## 四、NPC交互规则

NPC态度：友好→愿意帮助、中立→需说服、敌意→可能攻击、疯狂→不可预测

交涉：普通谈话无需检定，获取重要信息投【说服】，讨价还价困难难度，恐吓STR对抗
```

```text
# 完整游戏规则

## 一、检定系统（核心机制）

当玩家执行有风险的动作时，你需要进行"检定"来决定结果。

### 1.1 检定规则

检定流程：
1. 确定使用的技能或属性
2. 计算成功率（基于技能值或属性）
3. 从角色状态文本末尾的`骰值池`中取下一个d100值作为本次骰值（从左到右按顺序使用）
4. 骰值 ≤ 成功率 → 成功
5. 骰值 > 成功率 → 失败
6. 骰值 ≤ 成功率的1/5 → 大成功（极好效果）
7. 骰值 = 100 或 > 成功率的95% → 大失败（灾难性后果）

> ⚠️ 必须使用角色状态文本末尾`【本次骰值池】`中的d100值！每个检定消耗一个，用完回到开头循环。不允许自己编造骰值。

### 1.2 属性检定公式

属性值 × 5 = 基础成功率（例如STR 14 → 70%）

- STR 力量：推开重物、攀爬、近战伤害加成
- DEX 敏捷：闪避、潜行、平衡、先手顺序
- CON 体质：抵抗毒素/疾病、持续奔跑
- INT 智力：推理、解谜、记忆、理解复杂信息
- POW 意志：抵抗恐惧、说服他人、施展/抵抗法术

### 1.3 难度修正

- 有利条件（工具、环境优势）：+10% 到 +30%
- 不利条件（黑暗、紧张、受伤）：-10% 到 -30%
- 极度不利（狂风暴雨、重伤）：-40% 到 -60%

### 1.4 对抗检定

当玩家与NPC/怪物对抗时：双方各自投骰，都成功则成功率更高者胜，一方成功则成功者胜，都失败则维持现状。

## 二、战斗系统

战斗流程：先手判定(DEX对抗) → 攻击方投格斗 → 防守方可选闪避 → 命中则投伤害 → 描述结果

武器伤害：
- 徒手1d3、匕首1d4+1(可投掷)、撬棍1d6(可撬门)、棍棒1d6
- 猎刀1d6+1、手枪1d8(需火器技能)、猎枪1d10(装填慢)、临时武器1d4

伤害规则：
- HP ≤ 0：濒死，每回合继续损失1HP
- HP ≤ -max_hp：死亡
- 同伴可在3回合内急救稳定伤势

大成功（骰值≤成功率/5）：战斗双倍伤害/缴械，侦察发现额外信息，说服态度大幅提升
大失败（骰值=100或>成功率95%）：战斗武器脱手/自伤，潜行制造声响，交涉激怒对方

## 三、理智系统（SAN）

恐怖等级：
- 轻微恐怖：0/1d3（看到尸体、诡异声音）
- 中等恐怖：1/1d6（超自然现象、遭遇怪物）
- 强烈恐怖：1d3/1d10（直面神话生物、同伴惨死）
- 极致恐怖：1d6/2d10（目击不可名状之神、禁忌古籍）

临时疯狂（单次SAN损失≥5，投1d10）：
1-2:失忆 3-4:惊恐逃跑 5-6:暴力狂乱 7-8:幻觉 9-10:昏厥

SAN恢复：安全休息1d6、解谜推进1d3、镇静剂1d6、同伴安慰1d3、逃离危险1d3

## 四、NPC交互规则

NPC态度：友好→愿意帮助、中立→需说服、敌意→可能攻击、疯狂→不可预测

交涉：普通谈话无需检定，获取重要信息投【说服】，讨价还价困难难度，恐吓STR对抗
```

## 节点L：代码节点③ — 解析游戏输出

### 功能说明

这个节点解析LLM返回的文本，从中提取：
- HP变化量、SAN变化量
- 是否需要更新世界状态
- 世界更新参数（action、task_id等）

### 创建步骤

从**任意一个变量赋值节点**（如节点H1）底部拖线 → 选择"代码节点"。

然后从其他3个变量赋值节点（I1、J1、K1）的输出也连接到同一个代码节点。

**所有4个变量赋值节点汇聚到同一个解析节点，因为它们都往 llm_raw_output 写值，谁触发就读谁的数据。**

### 基本信息

| 字段 | 值 |
|------|-----|
| 节点名称 | `解析游戏输出` |
| 代码语言 | `Python3` |

### 输入变量

添加 **1个** 输入变量（从**会话变量**读取）：

| 变量名 | 来源 | 值 |
|--------|------|-----|
| `llm_output` | 会话变量 | `{{llm_raw_output}}` |

> **关键原理**：4个LLM各自执行完后，都把自己的输出文本写入同一个会话变量 `llm_raw_output`。解析节点读取这个会话变量，不管哪个LLM触发了当前路径，都能拿到正确的文本。这样就避免了"多个上游节点汇聚到同一个代码节点"的冲突问题。

### Python 代码

```python
import re
import json


def main(llm_output: str) -> dict:
    """
    解析LLM输出，提取状态变化、世界更新和PvP结果信息

    输入：LLM生成的文本（包含【状态变化】、【世界更新】、【PvP结果】标记）
    输出：结构化数据供后续节点使用
    """
    import random as _r

    # ---- 空值保护 ----
    if not isinstance(llm_output, str):
        llm_output = ""

    # ---- 默认值 ----
    hp_change = 0
    san_change = 0
    need_update = False
    world_json = {}

    # ---- 1. 解析【状态变化】行 ----
    status_match = re.search(
        r'【状态变化】[：:]\s*(.*?)(?=\n【|\n$|$)',
        llm_output,
        re.DOTALL
    )
    if status_match:
        status_line = status_match.group(1).strip()
        
        # 提取HP变化（支持数字或骰子表达式）
        hp_raw = re.search(r'HP:\s*([+-]?(?:\d+|(\d+)d(\d+)))', status_line)
        if hp_raw:
            hp_str = hp_raw.group(1)
            if 'd' in hp_str:
                n, m = int(hp_raw.group(2)), int(hp_raw.group(3))
                hp_change = sum(_r.randint(1, m) for _ in range(n))
                if hp_str.startswith('-'):
                    hp_change = -hp_change
            else:
                hp_change = int(hp_str)
        
        # 提取SAN变化
        san_raw = re.search(r'SAN:\s*([+-]?(?:\d+|(\d+)d(\d+)))', status_line)
        if san_raw:
            san_str = san_raw.group(1)
            if 'd' in san_str:
                n, m = int(san_raw.group(2)), int(san_raw.group(3))
                san_change = sum(_r.randint(1, m) for _ in range(n))
                if san_str.startswith('-'):
                    san_change = -san_change
            else:
                san_change = int(san_str)

    # ---- 2. 解析【世界更新】JSON ----
    # 先提取原始的JSON字符串（可能有嵌套大括号，所以用括号计数法）
    world_match = re.search(
        r'【世界更新】[：:]\s*(\{.*)',
        llm_output,
        re.DOTALL
    )
    if world_match:
        raw_json_str = world_match.group(1)
        # 括号计数法提取完整JSON
        depth = 0
        end_pos = 0
        for i, ch in enumerate(raw_json_str):
            if ch == '{':
                depth += 1
            elif ch == '}':
                depth -= 1
                if depth == 0:
                    end_pos = i + 1
                    break
        if end_pos > 0:
            try:
                world_json = json.loads(raw_json_str[:end_pos])
                if world_json.get("need_update"):
                    need_update = True
            except json.JSONDecodeError:
                world_json = {}

    # ---- 3. 从world_json提取字段 ----
    world_action = world_json.get("action", "")
    world_task_id = world_json.get("task_id", "")
    world_player = world_json.get("player", "")
    world_atmosphere_change = world_json.get("atmosphere_change", 0)
    msg = world_json.get("message", {})
    if isinstance(msg, dict):
        world_msg_location = msg.get("location", "")
        world_msg_content = msg.get("content", "")
        world_msg_player = msg.get("player", "")
        world_msg_player_status = msg.get("player_status", "")
    else:
        world_msg_location = ""
        world_msg_content = ""
        world_msg_player = ""
        world_msg_player_status = ""

    # ---- 4. 解析【数据更新】JSON（物品/位置/线索变化） ----
    data_location = ""
    data_sub_location = ""
    data_inventory = ""
    data_clues = ""
    data_raw_str = ""  # 保存原始JSON字符串，用于从story_text中剥离
    data_match = re.search(
        r'【数据更新】[：:]\s*(\{.*)',
        llm_output,
        re.DOTALL
    )
    if data_match:
        raw_data_str = data_match.group(1)
        depth = 0
        end_pos = 0
        for i, ch in enumerate(raw_data_str):
            if ch == '{':
                depth += 1
            elif ch == '}':
                depth -= 1
                if depth == 0:
                    end_pos = i + 1
                    break
        if end_pos > 0:
            data_raw_str = raw_data_str[:end_pos]
            try:
                data_json = json.loads(data_raw_str)
                data_location = data_json.get("location") or ""
                data_sub_location = data_json.get("sub_location") or ""
                data_inventory = data_json.get("inventory") or ""
                data_clues = data_json.get("clues") or ""
            except json.JSONDecodeError:
                pass

    # ---- 4b. 兜底：如果没找到【数据更新】标签，尝试从末尾提取JSON ----
    # LLM有时会忘记加标签，在剧情末尾输出裸JSON
    if not data_raw_str:
        end_json_match = re.search(r'\n(\{.*\})\s*$', llm_output, re.DOTALL)
        if end_json_match:
            try:
                fallback_json = json.loads(end_json_match.group(1))
                # 只有包含特定字段时才认作数据更新
                if any(k in fallback_json for k in ("location", "sub_location", "inventory", "clues")):
                    data_location = fallback_json.get("location") or ""
                    data_sub_location = fallback_json.get("sub_location") or ""
                    data_inventory = fallback_json.get("inventory") or ""
                    data_clues = fallback_json.get("clues") or ""
                    data_raw_str = end_json_match.group(1)
            except json.JSONDecodeError:
                pass

    # ---- 4c. 解析【PvP结果】JSON ----
    pvp_target = ""
    pvp_damage = 0
    pvp_hit = False
    pvp_is_kill = False
    pvp_match = re.search(
        r'【PvP结果】[：:]\s*(\{.*?\})',
        llm_output,
        re.DOTALL
    )
    if pvp_match:
        try:
            pvp_json = json.loads(pvp_match.group(1))
            pvp_target = pvp_json.get("target", "")
            pvp_damage = pvp_json.get("damage", 0)
            pvp_hit = pvp_json.get("hit", False)
            pvp_is_kill = pvp_json.get("is_kill", False)
        except json.JSONDecodeError:
            pass

    # ---- 5. 校验action是否合法 ----
    VALID_ACTIONS = ["update_atmosphere", "accept_task", "complete_task", "add_message"]
    if world_action and world_action not in VALID_ACTIONS:
        # LLM生成了未知action（如join_game），跳过世界更新
        need_update = False
        world_action = ""
        world_json = {}

    # ---- 5. 用自己的字段重新构建干净JSON ----
    clean_json = {"need_update": need_update}
    if world_action:
        clean_json["action"] = world_action
    if world_task_id:
        clean_json["task_id"] = world_task_id
    if world_player:
        clean_json["player"] = world_player
    if world_atmosphere_change:
        clean_json["atmosphere_change"] = world_atmosphere_change
    if world_msg_location or world_msg_content or world_msg_player:
        clean_json["message"] = {
            "location": world_msg_location,
            "content": world_msg_content,
            "player": world_msg_player,
            "player_status": world_msg_player_status,
        }
    world_json_str = json.dumps(clean_json)

    # ---- 6. 从story_clean中去掉【数据更新】/【世界更新】/【PvP结果】/末尾裸JSON ----
    story_clean = re.sub(
        r'\n?【(?:数据更新|世界更新|PvP结果)】[：:]\s*\{.*?\}\n?',
        '',
        llm_output,
        flags=re.DOTALL
    ).strip()
    # 如果有末尾裸JSON（来自4b兜底），也去掉
    if data_raw_str:
        story_clean = story_clean.replace(data_raw_str, "").strip()

    # ---- 7. 返回解析结果 ----
    return {
        "hp_change": hp_change,
        "san_change": san_change,
        "need_update": need_update,
        "world_json": world_json,
        "world_json_str": world_json_str,
        # 拍平字段（供HTTP节点直接引用）
        "world_action": world_action,
        "world_task_id": world_task_id,
        "world_player": world_player,
        "world_atmosphere_change": world_atmosphere_change,
        "world_msg_location": world_msg_location,
        "world_msg_content": world_msg_content,
        "world_msg_player": world_msg_player,
        "world_msg_player_status": world_msg_player_status,
        # 数据更新字段（位置/物品/线索）
        "data_location": data_location,
        "data_sub_location": data_sub_location,
        "data_inventory": data_inventory,
        "data_clues": data_clues,
        # PvP结果字段
        "pvp_target": pvp_target,
        "pvp_damage": pvp_damage,
        "pvp_hit": pvp_hit,
        "pvp_is_kill": pvp_is_kill,
        "story_clean": story_clean
    }
```

**操作方法**：

> 这样做避免了Dify的HTTP节点不支持Object嵌套字段的问题，也避免了空字段导致的400错误。

```python
def main(hp: int, san: int, hp_change: int, san_change: int,
         current_location: str, current_sub_location: str,
         current_inventory: str, current_clues: str,
         data_location: str, data_sub_location: str,
         data_inventory: str, data_clues: str) -> dict:
    """
    计算新的角色状态，合并数据更新
    data_xxx 来自LLM的【数据更新】，为空字符串=不修改
    """
    # ---- HP/SAN计算 ----
    new_hp = max(0, min(100, hp + hp_change))
    new_san = max(0, min(100, san + san_change))
    is_dead = new_hp <= 0
    is_crazy = new_san <= 0

    # ---- 数据合并：位置/物品/线索 ----
    # LLM在【数据更新】中输出的是完整值（非增量），
    # 有值就用LLM的，空字符串=不修改
    new_location = data_location if data_location else current_location
    new_sub_location = data_sub_location if data_sub_location else current_sub_location
    new_inventory = data_inventory if data_inventory else current_inventory
    new_clues = data_clues if data_clues else current_clues

    return {
        "new_hp": new_hp,
        "new_san": new_san,
        "is_dead": is_dead,
        "is_crazy": is_crazy,
        "new_location": new_location,
        "new_sub_location": new_sub_location,
        "new_inventory": new_inventory,
        "new_clues": new_clues,
    }
```

### 输出变量

| 输出变量 | 类型 | 说明 |
|---------|------|------|
| `new_hp` | Number | 计算后的HP值 |
| `new_san` | Number | 计算后的SAN值 |
| `is_dead` | Boolean | 是否死亡（HP≤0） |
| `is_crazy` | Boolean | 是否疯狂（SAN≤0） |
| `new_location` | String | 合并后的位置（来自数据更新或保持原值） |
| `new_sub_location` | String | 合并后的子场景 |
| `new_inventory` | String | 合并后的物品列表 |
| `new_clues` | String | 合并后的线索列表 |

### 连接

从节点O1底部输出连接到 **节点O2（变量赋值 — 更新角色状态）**。

## 附录：常见问题排查

### 问题1：代码节点报错 "变量未定义"

**原因**：输入变量的名字与代码中的参数名不一致。

**解决**：
1. 检查代码节点的输入变量列表，确保变量名和代码 `def main(...)` 中的参数名**一字不差**
2. 尤其注意下划线 `_`  和大小写
3. 删除不匹配的变量重新添加

### 问题2：LLM节点回复格式不对（没有【状态变化】或【世界更新】）

**原因**：LLM没有严格遵守输出格式。

**解决**：
1. 在系统提示词中强调输出格式要求
2. 降低温度值（从0.7降到0.5）
3. 在用户消息末尾追加提醒："请严格按照【输出格式】要求回复"
4. 考虑加入few-shot示例

### 问题3：路由不生效，总是走到同一个LLM

**原因**：IF/ELSE的条件配置错误或region值不匹配。

**解决**：
1. 在节点D（路由器）中加一行print调试：`print(f"current_location={current_location}, region={region}")`
2. 检查IF/ELSE条件中的值是否小写（`town` 不是 `Town` 或 `town区`）
3. 在chatflow配置说明.md中检查current_location的值是否在路由列表中

### 问题4：变量赋值节点报错

**原因**：Dify的变量赋值节点可能不支持算术表达式 `{{ hp + hp_change }}`。

**解决**：
1. 在节点L和节点O之间再加一个代码节点来计算新值
2. 或者在节点L的代码中直接返回新值（需要传入当前hp/san作为输入）

### 问题5：HTTP请求失败

**原因**：后端服务未运行、URL错误、网络不通。

**解决**：
1. 确认后端在运行：`http://192.168.88.100:8000/` 是否能访问
2. 在后端服务器上curl测试
3. 检查Dify服务器到后端服务器的网络连通性
4. 检查后端日志是否有错误

### 问题6：LLM输出被截断

**原因**：Max Token设置太小。

**解决**：
1. 增大Max Token（4000或更大）
2. 缩短系统提示词中的子场景描述
3. 检查模型是否支持更大的上下文窗口

### 问题7：dify中的变量引用路径不对

**原因**：Dify中不同版本的变量引用语法可能不同。

**解决**：
1. 在变量选择面板中手动选择，不要手工输入
2. 确认节点名称和输出变量名正确
3. 常见格式：`{{节点名.output.变量名}}` 或 `{{节点名.变量名}}`
