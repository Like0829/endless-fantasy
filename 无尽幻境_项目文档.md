# 《无尽幻境：AI游戏主持》项目文档

---

## 一、项目基本信息

| 项目名称 | 无尽幻境：AI游戏主持 |
|---------|---------------------|
| 项目周期 | 2天 |
| 参与人数 | 7人 |
| 技术平台 | Dify（Chatflow） |
| 项目类型 | AI驱动多人异步角色扮演游戏 |

---

## 二、项目概述

### 2.1 项目简介

一个由AI驱动的多人异步角色扮演游戏世界，玩家通过自然语言与AI游戏主持人（DM）交互，在克苏鲁/奇幻世界观下自由冒险。所有玩家共享同一个世界，每个人的选择都会影响其他玩家的游戏体验。

### 2.2 解决的问题

| 问题 | 解决方案 |
|------|---------|
| 传统跑团难约时间、难找主持人 | AI自动担任游戏主持人，随时可玩 |
| 现有AI文字游戏缺乏"世界感" | 共享世界状态库，玩家行为实时互相影响 |

### 2.3 核心亮点

**玩家行为塑造共享世界** —— 一个人熄灭的灯火，会成为另一个人永恒的黑暗。

---

## 三、游戏世界观与规则

### 3.1 背景设定

**主题**：调查迷雾镇

一座被诡异迷雾笼罩的小镇，所有镇民都消失了。玩家作为调查员陆续进入，揭开小镇背后的真相。

### 3.2 核心属性

| 属性 | 说明 | 归零后果 |
|------|------|---------|
| 理智值（SAN） | 角色精神状态 | 角色疯狂 |
| 生命值（HP） | 角色生命状态 | 角色死亡 |

### 3.3 基础动作判定

玩家描述行动 → AI根据难度进行"检定"（模拟投骰） → 给出成功/失败结果和剧情发展

## 四、系统架构

### 4.1 整体架构

```
[玩家] ---> [Dify Chatflow] ---> [外部世界后端 API] ---> [轻量数据库]
                                      |
                                      v
                                [全局世界状态]
```

### 4.2 各层职责

| 层级 | 职责 |
|------|------|
| **Dify层** | 对话管理、角色状态管理（会话变量）、剧情生成、规则判定 |
| **外部后端** | 提供读写"世界状态"的HTTP接口（Python） |
| **数据库** | 存储共享世界全局变量（MySQL） |

## 五、多人互动机制设计

### 5.1 世界悬赏板（共享任务）

**数据结构**：
```json
{
  "tasks": [
    {
      "id": "task_001",
      "name": "寻找医生的日记",
      "status": "pending/in_progress/completed",
      "player": null
    }
  ]
}
```

**交互流程**：
1. 玩家A接受任务 → 状态变为"进行中"
2. 玩家B想接同任务 → AI告知"已被接下"
3. 玩家A完成任务 → 状态变为"已完成"，记录完成者
4. 后续玩家询问 → AI告知已完成及结果

### 5.2 动态世界氛围（城镇氛围值）

**数据结构**：全局数值 `town_atmosphere`（0-100，初始50）

| 玩家行为 | 氛围变化 |
|---------|---------|
| 成功点亮灯塔 | +10 |
| 触发怪物警报 | -15 |

**剧情反馈**：
- 氛围 > 70："迷雾似乎淡了一些，空气中能闻到一丝海风的咸味，你感到希望尚存。"
- 氛围 < 30："整个小镇笼罩在令人窒息的黑暗中，到处都是怪物的低语，你的理智摇摇欲坠。"

### 5.3 异步留言/死亡墓碑

**数据结构**：
```json
{
  "messages": [
    {
      "location": "library",
      "content": "别相信镜子……",
      "player": "player_B",
      "status": "dead"
    }
  ]
}
```

**触发时机**：
- 角色死亡时 → AI强制让玩家留下遗言 → 写入API
- 其他玩家探索同地点 → AI检索并融入叙事

## 六、Dify Chatflow 详细设计

### 6.1 节点流程

```
开始节点
    ↓
会话变量读取（HP、SAN、位置、物品）
    ↓
HTTP请求节点（拉取全局世界状态）
    ↓
LLM节点1：意图分析与规则判定
    ↓
条件分支节点
    ├─ 世界状态变更 → HTTP请求节点（更新世界状态）
    └─ 无变更 ──────────↓
                         ↓
                   LLM节点2：叙事生成
                         ↓
                   会话变量更新
                         ↓
                   结束节点（输出回复）
```

### 6.2 各节点说明

| 节点 | 功能 |
|------|------|
| 会话变量读取 | 读取当前角色HP、SAN、位置、物品 |
| HTTP请求（读取） | 调用后端API获取世界状态 |
| LLM节点1 | 结合输入、角色状态、世界状态，判断检定需求 |
| 条件分支 | 判断是否需要更新世界状态 |
| HTTP请求（更新） | 调用后端API写入变更 |
| LLM节点2 | 生成沉浸式剧情描述 |
| 会话变量更新 | 写回HP、SAN等变化 |

## 七、外部后端 API 设计

### 7.1 接口列表

| 接口 | 方法 | 说明 |
|------|------|------|
| `/world-state` | GET | 返回完整世界状态 |
| `/update-world` | POST | 更新世界状态 |

### 7.2 接口详情

**GET /world-state**

返回：
```json
{
  "tasks": [...],
  "atmosphere": 50,
  "messages": [...]
}
```

**POST /update-world**

请求：
```json
{
  "action": "complete_task",
  "task_id": "1",
  "player": "A"
}
```
或
```json
{
  "action": "add_message",
  "location": "library",
  "content": "别照镜子！",
  "player": "B"
}
```

### 7.3 技术实现

- 后端框架：Python（FastAPI 或 Flask）
- 数据库：MySQL
- 部署：Railway / Render / 云服务器

## 十一、风险与应对

| 风险 | 概率 | 影响 | 应对策略 |
|------|------|------|---------|
| Dify HTTP节点超时 | 中 | 高 | 后端处理极简，只做简单读写，设置超时>30秒 |
| LLM忘记规则或状态 | 中 | 中 | 提示词中重复强调状态，用标签如`[WORLD_STATE]`分隔 |
| 多人同时更新数据冲突 | 低 | 低 | MySQL事务机制保证数据一致性 |
| 两天时间太紧 | 高 | 高 | 严格MVP原则：砍掉非核心功能，只保留1城镇、3任务、2种状态变化 |
